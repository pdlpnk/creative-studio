# Creative Analyzer v1

## Назначение

Creative Analyzer v1 — проектируемый инструмент подготовки Creative Concept Cards для изображений из `assets/references/`. Его задача — обнаруживать новые референсы, назначать им идентификаторы и формировать карточки в `brain/concepts/` через воспроизводимый и проверяемый процесс.

Версия 1 описывает архитектуру и контракт будущей реализации. Она не анализирует содержимое изображений, не подключает AI API, не создаёт GUI и не содержит исполняемого кода.

## Область работы

Входные данные:

- корневой каталог `assets/references/`;
- поддерживаемые изображения: `.jpg`, `.jpeg`, `.png`, `.webp`;
- расположение файла, из которого можно получить предварительные `geo` и `funnel`;
- существующие Concept Cards в `brain/concepts/`.

Результат будущего процесса:

- уникальный ID формата `CC-0001` для нового концепта;
- Markdown Concept Card по структуре `brain/concepts/template.md`;
- карточка, сохранённая как `brain/concepts/CC-0001.md`;
- явный статус обработки исходного изображения;
- журнал ошибок и причин, по которым карточка не была создана.

Само изображение остаётся в reference library и не изменяется, не сжимается и не переименовывается анализатором.

## Архитектура

Будущий анализатор разделяется на независимые компоненты.

### 1. Reference Scanner

Сканирует `assets/references/` рекурсивно и выбирает только поддерживаемые изображения. Игнорирует `.gitkeep`, README, видео, архивы, временные и системные файлы.

Для каждого файла scanner собирает безопасные технические метаданные:

- относительный путь;
- расширение;
- размер файла;
- время изменения;
- контрольную сумму содержимого;
- предварительные `geo` и `funnel`, если они однозначно следуют из пути.

Контрольная сумма нужна для поиска дубликатов и устойчивого распознавания уже известных файлов независимо от имени.

### 2. Processing Registry

Хранит состояние обработки отдельно от изображений. Реализация должна использовать машиночитаемый реестр или manifest, не добавляя метки в бинарные файлы и не меняя их имена.

Минимальные состояния:

- `unprocessed` — новое изображение найдено, карточка ещё не создана;
- `pending_review` — черновик карточки создан и ожидает проверки;
- `processed` — карточка подтверждена и связана с изображением;
- `failed` — обработка не завершена; сохранена понятная причина.

При первом обнаружении новый уникальный файл получает состояние `unprocessed`. Повторный запуск не должен создавать дубликат карточки для той же контрольной суммы.

Точный формат и место хранения registry будут выбраны на этапе реализации. Registry не должен содержать секреты или персональные данные.

### 3. ID Allocator

Читает существующие карточки в `brain/concepts/`, проверяет их YAML front matter и определяет следующий свободный ID `CC-NNNN`.

Требования к назначению ID:

- ID уникален во всём репозитории;
- уже выданные ID не переиспользуются;
- имя файла совпадает с ID: `CC-0001.md`;
- операция резервирования должна быть атомарной, чтобы параллельные запуски не получили один ID;
- существующие карточки и примеры не перезаписываются.

### 4. Metadata Extractor

Без AI компонент может заполнить только данные, которые надёжно выводятся из пути и технических метаданных:

- `geo` из каталогов `tr` или `az`;
- `funnel` из каталогов `registration` или `lead`;
- `format`, если размеры изображения были безопасно прочитаны;
- `asset_url` как относительную ссылку на исходный файл;
- технические поля происхождения и времени обработки.

Смысловые поля — `theme`, `style`, `emotion`, `objects`, `composition`, `headline`, `visual_brief` и `hypothesis` — не должны угадываться без анализа. До подключения анализатора изображения они остаются пустыми и требуют ручного заполнения либо карточка сохраняется как черновик `pending_review`.

### 5. Analysis Provider

Предусматривается интерфейс поставщика анализа, отделённый от scanner, ID allocator и writer.

В Creative Analyzer v1 используется только заглушка без сетевых вызовов. Позже через этот интерфейс можно будет подключить OpenAI API, не меняя остальной pipeline.

Будущий provider должен:

- принимать изображение и строго заданную схему результата;
- возвращать только структурированные поля Concept Card;
- не назначать ID и не записывать файлы самостоятельно;
- явно сообщать о неполном или недостоверном результате;
- поддерживать версию модели и версию инструкции для воспроизводимости;
- проходить проверку compliance до сохранения результата.

API-ключи в репозитории, карточках, registry или логах хранить запрещено.

### 6. Concept Card Builder

Объединяет ID, технические метаданные и результат provider в модель карточки. За основу берётся `brain/concepts/template.md`.

Builder должен:

- сохранять все обязательные поля шаблона;
- использовать `format: 1080x1080` только для соответствующих изображений;
- устанавливать безопасный начальный `status: idea`;
- связывать карточку с исходным изображением через `asset_url`;
- не присваивать `winner` или `loser` без результата тестирования;
- помечать незаполненные смысловые поля для ручной проверки, а не придумывать значения.

### 7. Validator

До записи проверяет:

- уникальность и формат ID;
- допустимость `geo`, `funnel`, `format` и `status`;
- наличие обязательных полей;
- существование указанного asset;
- отсутствие второй карточки с той же контрольной суммой изображения;
- корректность YAML front matter;
- отсутствие неподтверждённых обещаний, гарантий выигрыша, ложных отзывов и скрытых условий в создаваемом тексте.

Если проверка не пройдена, файл карточки не создаётся, а registry получает состояние `failed` с причиной.

### 8. Markdown Writer

Формирует Concept Card в YAML front matter + Markdown и записывает её в `brain/concepts/<concept_id>.md`.

Запись должна быть атомарной: сначала формируется временный результат, затем после успешной валидации он перемещается на конечный путь. Writer никогда не перезаписывает существующую карточку без явного отдельного режима обновления.

### 9. Reporter

По завершении показывает сводку без GUI:

- сколько файлов найдено;
- сколько новых файлов помечено `unprocessed`;
- сколько черновиков создано;
- сколько файлов пропущено как уже обработанные или дублирующиеся;
- сколько ошибок возникло и где найти их причины.

## Поток обработки

```text
assets/references/
        ↓
Reference Scanner
        ↓
Processing Registry ── already processed / duplicate → skip
        ↓ new: unprocessed
ID Allocator
        ↓
Metadata Extractor
        ↓
Analysis Provider (v1: no AI, future: OpenAI adapter)
        ↓
Concept Card Builder
        ↓
Validator ── invalid → failed
        ↓ valid
Markdown Writer
        ↓
brain/concepts/CC-NNNN.md
        ↓
pending_review → human review → processed
```

## Идемпотентность и безопасность

- Повторный запуск с тем же набором файлов не создаёт новые карточки.
- Изображения считаются одинаковыми по контрольной сумме, а не только по имени.
- Пользовательские изображения никогда не изменяются и не удаляются.
- Существующие Concept Cards не перезаписываются.
- Ошибка одного файла не останавливает учёт остальных файлов.
- Частично записанная карточка не должна оставаться в `brain/concepts/`.
- Все автоматически полученные смысловые данные требуют человеческой проверки перед производственным использованием.

## Этапы будущей реализации

1. Утвердить формат Processing Registry и правила его версионирования.
2. Реализовать scanner, checksum и обнаружение новых файлов без анализа содержимого.
3. Реализовать безопасное назначение ID, builder, validator и Markdown writer.
4. Добавить dry-run и отчёт о планируемых изменениях.
5. Проверить pipeline на локальных reference-файлах без сетевых вызовов.
6. Отдельным этапом определить контракт OpenAI provider, требования к structured output и обработку ошибок API.
7. Подключать OpenAI API только после отдельного согласования, не сохраняя ключи в репозитории.

## Что не входит в v1

- анализ содержимого изображений;
- OpenAI API или другие внешние API;
- Python AI и локальные AI-модели;
- генерация изображений или промптов;
- Google Sheets интеграция;
- GUI;
- изменение или удаление reference-файлов;
- автоматическая публикация карточек без проверки.
